### Issues found and FIXED this session (implementation-level, no founder input)

1. **HTML injection in purchase-receipt email (MEDIUM, fixed).** `lib/email/receipt.ts` interpolated user-supplied `buyerName` (and `packageName`/`orderId`) into the HTML body unescaped — a content-injection/phishing vector since name is user-editable in onboarding. Fixed with an `escapeHtml()` guard on all interpolated fields; added a regression test in `tests/email.test.ts`. With this, **parked Tier-A ticket-10-email-receipt passes Fable review** (call-site placement was already correct: post-commit, fail-safe, idempotent).

### Issues verified as NOT issues (agent claims rejected on inspection)

- Admin tables at 320px: already wrapped in `overflow-x-auto` with sensible `min-w` — correct pattern, no fix needed.
- "Missing aria-labels for monograms": no such component exists in the codebase.

### Known tech debt (already recorded in GPS — confirmed accurate, do not lose)

1. Shared segmented OTP input + resend timer + failure-path UX (checkout + login) — scheduled for the live-provider pass.
2. PII encryption module (`AES-256-GCM`) not yet implemented — schema slots ready; **must exist before KYC ships (Phase 4)**. Env validation for `PII_ENCRYPTION_KEY` already enforced in production.
3. CSP header — deferred post-launch (needs co-design with Razorpay/Supabase/Stream/PostHog origins). Acceptable.
4. Dashboard/admin `loading.tsx` skeletons — add during Phase 2/4 module work (pages are IN DEVELOPMENT; building them now would preempt GPS-M2 spec execution).
5. Client-side phone Zod validation parity on checkout (server already validates) — fold into OTP UX debt item.

## 5. Phase 4 — Remaining Execution Roadmap

Order follows DR-026 (module-by-module to 100%). Phases are gates, not dates.

**P2 — LMS to 100% (NOW · Launch Critical).** Objective: complete GPS-M2 spec — dashboard, course player (mock video), progress, profile, my-courses, empty/loading states. Dependencies: GPS-M2 spec (exists), Stream adapter (done). Complexity: medium. Outcome: a buyer can learn end-to-end with mock video; real video slots in when content lands. Blocks launch: yes.

**P3 — Affiliate to 100% (Revenue Critical).** Objective: referral tree UI, wallet UI (held vs available), share tooling; payouts stay OFF until D-01. Dependencies: ledger (done), D-01 for activation only. Complexity: medium. Blocks launch: no (earning UI can ship dark); blocks revenue-model activation: yes.

**P4 — Admin + KYC (Revenue Critical for payouts).** Objective: admin dashboard, settings with `AFFILIATE_PAYOUTS_ENABLED`, PII encryption lib (AES-256-GCM), KYC intake, withdrawal queue. Dependencies: D-01, PII key. Complexity: medium-high (money-adjacent, Tier A throughout). Blocks launch: no; blocks payouts: yes.

**P-Launch — Go-Live pass (Launch Critical, parallelizable with P2/P3).** Objective: live providers (Razorpay, MSG91, Stream, Resend, PostHog, domain), legal pages content in, ₹1 live purchase test, PITR enabled, OTP UX debt pass, Lighthouse re-run, CSP header. Dependencies: founder accounts + counsel. Complexity: low-medium engineering, high coordination. Blocks launch: yes — this IS the launch gate.

**P5 — Premium Experience (Post-Launch / Future Vision).** Guru Hinglish tutor (RAG over transcripts, D-29 guardrails, cost caps), gamification, quizzes, WhatsApp layer, PWA, certificates verify page (Slice 1.5 fast-follow). Dependencies: recorded content, real users. Blocks launch: no.
